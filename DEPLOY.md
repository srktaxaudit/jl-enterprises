# Deploying the JL backend to Render

There is now **one** backend — [`ecommerce-backend/`](ecommerce-backend/) — which
serves everything under `/api/v1/**` (auth, catalog, cart, orders, payments,
coupons, reviews, admin, audit). The static `admin.html` pages call this same
API (bearer-token `/api/v1/auth`).

It reads Render's injected `PORT`, exposes `/actuator/health`, and ships a
multi-stage `Dockerfile`.

| Service | Folder | Database |
|---|---|---|
| `jl-enterprises-api` | `ecommerce-backend/` | **Supabase Postgres** + Redis `jl-ecom-redis` |

> **Database = Supabase.** The backend auto-creates its schema on first boot
> (Hibernate), so you run **no SQL** in the Supabase editor. Just supply the
> connection string. Use Supabase → Project Settings → Database → **Session pooler**
> (IPv4 + prepared-statement compatible; do NOT use the Transaction pooler on 6543).

---

## Option A — One-click Blueprint (recommended)

1. Create a **Supabase** project (or reuse one). Copy the **Session pooler** connection
   info from Project Settings → Database.
2. Push this repo to GitHub/GitLab.
3. Render → **New → Blueprint** → select the repo. Render reads [`render.yaml`](render.yaml)
   and creates `jl-ecom-redis` and `jl-enterprises-api`.
4. Fill the values marked `sync: false` — the three `DB_*` vars from Supabase (see the env
   table), `CORS_ORIGINS`, and `APP_BOOTSTRAP_ADMIN_PASSWORD` — then **Apply**.
   (`REDIS_HOST`, `REDIS_PORT` and `JWT_SECRET` are wired automatically.)
5. First boot: Hibernate creates all tables in Supabase. Verify in the Supabase Table Editor.

## Option B — Manual (single Web Service)

**New → Web Service** → connect repo → **Root Directory** `ecommerce-backend` →
**Runtime: Docker** → Health check path `/actuator/health/liveness` (matches `render.yaml`)
→ add the env vars below → Create.
Point the `DB_*` vars at your Supabase Session pooler and (optionally) add a Key Value store.

## Env vars — `jl-enterprises-api`

| Key | Value |
|---|---|
| `DB_URL` | `jdbc:postgresql://aws-0-<region>.pooler.supabase.com:5432/postgres?sslmode=require` (Supabase Session pooler) |
| `DB_USER` | `postgres.<project-ref>` (Supabase Session pooler user) |
| `DB_PASSWORD` | your Supabase database password |
| `DB_POOL_MAX` | `5` (Supabase free tier has limited connections) |
| `REDIS_HOST` / `REDIS_PORT` | from `jl-ecom-redis` (auto-wired by the blueprint) |
| `SPRING_PROFILES_ACTIVE` | **`prod`** — activates the production config checks (see *Production safety checks* below) |
| `JWT_SECRET` | auto-generated by the blueprint (≥32 bytes), or set your own strong random value |
| `CORS_ORIGINS` | the **exact** origins allowed to call the API, comma-separated — **must include wherever the static admin/store is hosted**, e.g. `https://jlstores.in,https://www.jlstores.in`. No wildcards, no localhost |
| `APP_SECURITY_CORS_ALLOW_VERCEL_PREVIEWS` | `false` — leave off in production (see *Vercel preview origins* below) |
| `APP_BOOTSTRAP_ADMIN_PASSWORD` | initial SUPER_ADMIN password — strong & unique (≥12 chars, not a known default) — **change after first login** |
| `MAIL_HOST` / `MAIL_PORT` / `MAIL_USERNAME` / `MAIL_PASSWORD` | SMTP — needed for password-reset & verification emails to actually send (e.g. Gmail app-password or Resend SMTP) |
| `MAIL_SMTP_AUTH` / `MAIL_STARTTLS` | `true` / `true` when using a real SMTP host |
| `PASSWORD_RESET_URL` | `https://<your-vercel-domain>/reset-password.html` (where the reset email links) |
| `SUPABASE_URL` | `https://<project-ref>.supabase.co` — for product **image uploads** |
| `SUPABASE_SERVICE_KEY` | Supabase **service_role** key (server-side only — never expose to the browser) |
| `SUPABASE_BUCKET` | `product-images` (a **public** Storage bucket you create) |
| `RAZORPAY_KEY_ID` / `RAZORPAY_KEY_SECRET` | Razorpay keys for **online payments** (test keys first, then live). Blank = COD-only |
| `JPA_DDL_AUTO` | `update` for first boot (see below) |
| `FLYWAY_ENABLED` | `false` for first boot (see below) |

After deploy:
- Swagger: `https://<service>.onrender.com/swagger-ui.html`
- A SUPER_ADMIN is seeded (`admin@jlenterprises.in` / `APP_BOOTSTRAP_ADMIN_PASSWORD`).

## Point the static admin at the deployed API
In [`frontend/config.js`](frontend/config.js), set the production URL (this single line is
loaded before `store.js` / `admin.js` on every page):
```js
window.JL_API_BASE = "https://jl-enterprises-api.onrender.com";   // your Render service URL
```
Then add that static site's origin to `CORS_ORIGINS`. Log in at `admin-login.html`
with the seeded SUPER_ADMIN, and manage staff/roles from `admin-team.html`.

## Frontend on Vercel (the `frontend/` folder is the store)

The deployed storefront + admin is the static site in **[`frontend/`](frontend/)**.
In the Vercel project, set **Settings → Build & Output → Root Directory = `frontend`**
(Framework preset **Other**, no build command; the repo root holds no app — the old
Next.js prototype was removed from git). Add your exact Vercel domain to `CORS_ORIGINS`.

### Vercel preview origins

Every Vercel preview build gets its own `*.vercel.app` subdomain, so preview URLs are
**not** covered by `CORS_ORIGINS` and cannot call the API by default. That is deliberate:
allowing them means allowing the pattern `https://*.vercel.app`, which matches **every**
site on `vercel.app` — not just ours — so any of them could call this API from a browser.

To test a preview build against the live API, enable it **temporarily**:

1. Render → the `jl-enterprises-api` service → **Environment** →
   set `APP_SECURITY_CORS_ALLOW_VERCEL_PREVIEWS` to `true` → save (the service redeploys).
   Each boot then logs a loud warning that previews are enabled.
2. Do the preview testing.
3. Set it back to **`false`** and save. Do not leave it on.

Prefer promoting the build to a real domain listed in `CORS_ORIGINS` over enabling this.

## Production safety checks

With `SPRING_PROFILES_ACTIVE=prod`, the app **refuses to boot** on unsafe configuration
rather than running insecurely (`StartupConfigValidator`). It fails, naming the variable
to fix but **never printing the value**, when:

- `JWT_SECRET` is missing, is the built-in development default, or is shorter than 32 bytes
- `DB_PASSWORD` is missing/blank or is the development default
- `CORS_ORIGINS` is missing, contains a wildcard (`*`), or contains a localhost origin

An unsafe `APP_BOOTSTRAP_ADMIN_PASSWORD` (missing, shorter than 12 characters, or a
well-known weak value) logs a **loud warning on every boot** but does not block startup:
the value is only ever used when seeding a brand-new empty database, and that seeding
path separately refuses to run with a default value. Set a strong value anyway.

As a safety net these checks also run without the `prod` profile whenever the datasource
is not local, so a deployment that forgot `SPRING_PROFILES_ACTIVE` is still covered.
Local development (localhost DB, no `prod` profile) is unaffected — the built-in defaults
keep working and only log warnings. See [`.env.example`](.env.example) for the full list.

## Supabase Storage (product images)

1. Supabase → **Storage → New bucket** → name `product-images` → **Public**.
2. Project Settings → **API** → copy the **Project URL** and the **service_role** key.
3. Set `SUPABASE_URL`, `SUPABASE_SERVICE_KEY`, `SUPABASE_BUCKET=product-images` on Render.
4. In admin **Products**, create/edit a product and use **Upload image** — the photo then
   shows automatically on the storefront (the ★ image is the primary/thumbnail).

## Razorpay (online payments)

1. Create a Razorpay account → **API Keys** → generate **test** keys first.
2. Set `RAZORPAY_KEY_ID` / `RAZORPAY_KEY_SECRET` on Render.
3. At checkout, "Pay Online" opens Razorpay Checkout; the backend verifies the signature
   before marking the order paid. COD works with no keys. Switch to live keys when ready.

### Payment webhook (strongly recommended once online payments are live)

The browser confirm callback can be lost (customer's phone dies right after paying).
The webhook is the durable path — with it, a captured payment always settles the order,
and the abandoned-order sweeper double-checks Razorpay before ever cancelling.

1. Invent a strong random secret (e.g. `openssl rand -hex 24`).
2. Razorpay dashboard → **Settings → Webhooks → Add New Webhook**:
   - URL: `https://<service>.onrender.com/api/v1/webhooks/razorpay`
   - Secret: the value from step 1
   - Active events: **payment.captured**
3. Set `RAZORPAY_WEBHOOK_SECRET` on Render to the same value. Until it is set, the
   endpoint rejects all events (the browser confirm flow keeps working regardless).

### Refunds

Refunds now move REAL money: cancelling/refunding a paid Razorpay order calls the
Razorpay refund API first and only updates stock/books/status if the gateway refund
succeeds (a failure shows a clear error and changes nothing). COD refunds are recorded
as `MANUAL` — staff hand the cash back. If a payment is captured for an order that was
already cancelled, it is auto-refunded and admins are notified.

## Keep the free tier awake (stops cold-start failures)

**Already automated:** [`.github/workflows/keepalive.yml`](.github/workflows/keepalive.yml)
pings `/actuator/health/liveness` every ~12 minutes from GitHub Actions — nothing to set
up. (An external monitor like cron-job.org/UptimeRobot is only a fallback if GitHub
Actions is ever disabled on the repo.) The storefront and admin also retry cold starts.

> Monitoring note: check `/actuator/health/liveness` (or `/readiness`), NOT the plain
> `/actuator/health` aggregate — an optional component (e.g. SMTP) reporting DOWN turns
> the aggregate 503 while the store serves customers fine.

## Before you call it production

- **Freeze the schema into Flyway.** First boot uses `JPA_DDL_AUTO=update` (Hibernate
  creates the tables). Once stable: dump that schema into
  `ecommerce-backend/src/main/resources/db/migration/V2__schema.sql`, then set
  `JPA_DDL_AUTO=validate` and `FLYWAY_ENABLED=true` so Flyway owns the schema.
- **Free plan caveats:** free web services sleep after inactivity (cold starts) and
  free Postgres/Redis have limits — fine for testing, upgrade for real traffic.
- **Payments:** **COD** and **Razorpay** are implemented (set the Razorpay keys). The
  Stripe/PayPal strategies remain stubs — add real keys + SDK calls before using them.
- **Secrets:** never commit real values; set them only in the Render dashboard.
